# 创建扉页测评

把一个主题变成可直接导入「扉页」的测评 JSON，并在交付前完成结构与评分覆盖校验。

## 工作流

1. 获取主题。主题缺失时只追问主题；其余参数默认使用 10 道题、4 个选项、6 个结果。
2. 读取 [references/schema.md](references/schema.md)，遵守字段、内容和评分约束。
3. 先设计互相可区分的结果原型，再建立每个结果的主要得分机会矩阵。
4. 写具体场景题，不直接询问抽象性格。让每个选项都可被体面地选择，不设置明显“正确答案”。
5. 只在能确认出处时填写文学引文；不确定时把 `quote` 写成空字符串，禁止编造。
6. 把完整对象保存为 UTF-8 `.json` 文件。若用户只要内容，也必须保持标准 JSON，不添加注释和尾逗号。
7. 运行校验器，修复错误后再次运行，直到退出码为 0：

```bash
node scripts/validate_quiz.mjs /absolute/path/to/quiz.json
```

8. 需要直接分享时生成链接：

```bash
node scripts/validate_quiz.mjs /absolute/path/to/quiz.json \
  --share-url https://pjbian1.github.io/feiye/
```

9. 交付 JSON 文件、校验摘要和分享链接。提醒用户先自己试玩一遍，尤其核实文学出处和引文。

## 质量门槛

- 使用 8—12 道题、每题 4 个选项、4—8 个结果，除非用户明确指定其他数量。
- 让每个结果至少获得 4 次主要得分机会，且各结果机会数尽量接近。
- 让主要倾向记 2 分、次要倾向记 1 分；每个选项最多指向两个结果。
- 把结果解读控制在 120—180 字：肯定独特之处，点破一个软肋，用一句值得截图的话收尾。
- 不输出心理诊断、医学结论、歧视性标签或必然命运判断。
- 不跳过校验器，也不把“看起来像合法 JSON”当作通过。

## 失败处理

- 校验器报 `errors`：修改 JSON 并重跑。
- 校验器只有 `warnings`：优先修复评分覆盖；用户明确接受时才带警告交付。
- 分享链接超过 8,000 字符：缩短结果解读或减少冗余字段，再重新生成。
- 网站地址不可用：仍交付已校验 JSON，不虚构已发布链接。
